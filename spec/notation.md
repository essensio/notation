# Essensio — система типов

Тип — это **домен**: именованное множество допустимых значений вместе с операциями над ними. Множество говорит, *что* вообще может стоять в ячейке: «Число» — все числа, «Строка» — все тексты, «Дата» — все календарные даты. Операции говорят, *что* с этими значениями можно делать. Это понятие из реляционной модели: домен у Кодда, «тип = домен» у Дейта и Дарвена.

## Системные и пользовательские домены

Домены бывают двух родов. **Системный** задаёт движок программным алгоритмом — потому что его устройство не свести к ограничению: какие дни существуют у «Даты» (високосы, длины месяцев), считает код, а не булево условие над строкой. Системные домены: **Строка** (она же корень и цель сериализации), **Число**, **Булево**, **Дата**, **Время**, **UUID** (значения порождает движок — это идентификаторы записей `#`). Особняком — вырожденный домен **Пусто**: ровно одно значение, литерал `null`.

**Пользовательский** домен — существующий тип плюс **булево выражение**, сужающее его множество: `Положительное = Число | _ > 0`; `Процент = Число | _ >= 0 and _ <= 100`; `Почта = Строка | _ ~ r".+@.+"`. Здесь `_` обозначает проверяемое значение, а выражение пишется в операциях родителя (у чисел — сравнение, у строк — членство в множестве строк). Это specialization by constraint у Дейта и Дарвена: подтип = значения родителя, удовлетворяющие предикату.

Тип можно собрать и как **произведение** нескольких типов — но всегда с **именованными** компонентами, то есть как **кортеж** (`Точка = {x: Число, y: Число}`, ниже). Тогда операции наследуются от компонент, а ограничение опирается уже не на одно значение, а на несколько компонент.

## Операции

**Операция** — это функция над значениями: берёт одно или несколько значений известных типов и возвращает значение какого-то типа. У операции есть **сигнатура** — типы входов и тип результата: `+ : Число, Число -> Число`; `> : Число, Число -> Булево`; `len : Строка -> Число`. Каждый тип несёт свои операции; равенство `= : T, T -> Булево` есть у каждого типа. Среди операций каждого типа — **сериализация**: `записать` (значение → строка-литерал) и `разобрать` (строка → значение, с проверкой домена), связанные тождеством `разобрать(записать(v)) = v`.

Откуда операции: у **системного** типа встроены движком; у **ограничения** — те же, что у родителя (`Положительное` умеет всё, что `Число`, ведь его значения — подмножество чисел); у **произведения** — доступ к компонентам плюс операции, поднятые от компонент.

Каталог системных типов (из этих операций складываются выражения):

- **Число**: `+ - * /` (-> Число), `< <= > >=` (-> Булево);
- **Строка**: `+` склейка (-> Строка), `len` (-> Число), `< <= > >=` лексикографически (-> Булево); правый операнд членства `~` для строк может задаваться регэкспом `r"…"` — регулярный язык как множество строк;
- **Булево**: `not`, `and`, `or` (-> Булево);
- **Дата**: `< <= > >=` (-> Булево), `-` разность (-> Число дней);
- **Время**: `< <= > >=` (-> Булево), `-` (-> Число минут);
- **UUID**: только равенство;
- у **каждого** типа есть `=`, `!=` и членство `~ : T, T[] -> Булево`.

**Запись операторов.** Канон — клавиатурные символы (`= != < > <= >= + - * / ~ | # [ ] { } ( )`); математические глифы (`≤ ≥ ≠ ∈ × →`) допустимы лишь как необязательная экранная подсветка и для ввода не требуются.

## Выражения

Выражение строится из форм: **литерал** (готовое значение — `5`, `Дата("…")`), **ссылка** на значение (имя поля или компоненты — `количество`, `a.x`), **применение операции** (инфиксной `x + 1`, `a != b` или именованной `len(s)`) и **скобки** для группировки. Композиция: результат одной операции подаётся на вход другой. Каждое выражение **имеет тип** — тип своего результата, проверяемый заранее по сигнатурам: `(_ > 0) and (_ < 100)` — два сравнения, связанные `and` (-> Булево); а `5 + "текст"` не соберётся. Так круг замыкается: операции типа → выражения → новые типы.

**Булево** выражение (результат — `Булево`) применяется в двух местах: как **ограничение** подтипа (`Число | _ > 0`) и как условие **выборки** в реляционной алгебре (`σ` оставляет кортежи, на которых оно истинно). Язык выражений один — потому он и принадлежит системе типов, а не заводится в каждом месте отдельно.

**Переменные.** Выражение пишут не в пустоте — у него есть **контекст**, связывающий имена с типами. В ограничении подтипа-скаляра доступен `_` (проверяемое значение; тип — родитель). В ограничении кортежа — имена компонент (`| ширина >= высота`). В выборке `σ` — поля текущего кортежа (`σ[количество > 1]`). Имя и его тип берёт контекст; поэтому выражение проверяется заранее, без догадок.

**Логические связки и булевы литералы** — латиницей, не русскими словами: `and` / `or` / `not`, `true` / `false`. Имена типов, полей и операций остаются на языке домена.

Формальная грамматика — в разделе «Сводная грамматика»; приоритет задаётся иерархией продукций, от слабого (`or`) к крепкому (доступ `.`, применение). Три платформенные точки: `~` стоит на уровне сравнений; сравнения **не цепляются** (`0 <= _ <= 100` пишем как `_ >= 0 and _ <= 100`); `not` — на булевом уровне (слабее сравнения, как в Python). Форма `имя(…)` — применение операции, если `имя` операция, и селектор-литерал, если `имя` тип; различается резолвингом имени, не грамматикой.

## Кортеж

**Кортеж** — это произведение типов с **именованными компонентами**: `Точка = {x: Число, y: Число}`, а значение — `Точка{x: 3, y: 5}`. Компоненты сами имеют типы — любые, включая другие кортежи, — поэтому из кортежей складываются вложенные структуры.

**Операции кортежа.** Базовая — **доступ к компоненте** (`a.x`): из `Точки` берутся `x` и `y` (каждая — `Число` со своими операциями). Равенство по значению поднимается от компонент (два кортежа равны, когда равны все компоненты); поверх можно объявить и свои операции — скажем, `расстояние : Точка, Точка -> Число`.

**Ограничение кортежа** может опираться на несколько компонент сразу: `Отрезок = {a: Точка, b: Точка} | a != b`; `Прямоугольник = {ширина: Положительное, высота: Положительное} | ширина >= высота`. Выражение видит все компоненты и связывает их — чего ограничение над одним значением не может. Так кортеж с ограничением складывает из системных типов составные доменные типы.

## Отношение

**Отношение** — это тип, чьё **значение само есть множество**: набор кортежей одной шапки, без порядка. Если кортеж описывает одну запись, то отношение над ним — это много записей как одно значение. В нотации тип-отношение записывается `Кортеж[]`, а значение — список кортежей `[ {…}, {…} ]`.

**Операции — реляционная алгебра**: **выборка** (по условию), **проекция** (часть компонент), **соединение** (по совпадению), **переименование**, и множественные — объединение, пересечение, разность. Все **замкнуты**: результат каждой — снова отношение, поэтому они вкладываются друг в друга и собираются в запрос. Два отношения равны, когда совпадают как наборы записей.

**Идентичность.** Кортеж, попадая в отношение, становится **записью** и получает идентификатор `#: UUID`, порождаемый движком (у value-кортежа как значения поля его нет). Отсюда **два равенства**: *по значению* — совпадают компоненты (та же data) — и *по идентичности* — совпадает `#` (та же запись). Две записи могут быть равны по значению, но различны по идентичности; этим запись и становится **сущностью**. `#` — **суррогатный ключ** (значение, а не указатель, поэтому Информационный принцип TTM держится), и его неизменность снимает проблему изменчивых естественных ключей — те выражаем ограничением уникальности поверх полей.

## Объявление типов

Новый тип объявляют **одной декларацией** `Имя = T`, где `T` — **выражение-тип**, и это **данные** модели, а не код движка. Выражения-типы образуют маленькую **замкнутую** алгебру: `T` подставляется всюду, где ожидается тип (тип поля, элемент отношения, родитель ограничения), — поэтому пять форм ниже свободно вкладываются друг в друга.

- **имя типа** — `Число`, `Заказ`: системный или ранее объявленный тип;
- **ограничение-подтип** — `T | b`, где `b` — булево выражение; единственный элемент скаляра обозначается `_`, у кортежа доступны имена полей: `Положительное = Число | _ > 0`; `Категория = Строка | _ ~ ["грузчик", "кассир", "комплектовщик"]`;
- **кортеж** — `{ имя: T, … }`: `Заказ = {объект: Адрес, дата: Дата, количество: Количество}`; тип компоненты — любой `T`, включая кортеж или отношение (компонента-отношение — это TTM-овский **RVA**: `Клиент = {имя: Строка, заказы: Заказ[]}`);
- **отношение** — `T[]`: `Заказы = Заказ[]`;
- **ссылка (внешний ключ)** — `#T` на месте типа поля: `заказ: #Заказ`; хранит значение `#` целевой записи — ссылка по значению, а не указатель.

Системные типы (`Строка … UUID`) встроены — их не объявляют. Всё прочее — комбинация форм выше: «новая возможность = новый тип, а не новый код». Раз тип — это данные, движок резолвит имя в определение, а само определение можно показать, проверить, сохранить, как любые данные.

Формальная грамматика — в разделе «Сводная грамматика».

`предикат` — выражение типа `Булево` (грамматика — в разделе «Выражения»): использует операторы `=` `!=` `<` `>` `<=` `>=` `~` и связки `not` / `and` / `or`, а единственный элемент скаляра в нём обозначается `_`. Три зашитых правила:

1. `|` — только на верхнем уровне типа; уточнение внутри поля — через скобки (`{цена: (Число | _ > 0)}`), а ограничение кортежа стоит после `}` (`{…} | b`);
2. `#` применяется только к **имени** (`#Заказ`) — на безымянный кортеж/отношение сослаться нельзя;
3. безымянная вложенность `{…}` / `T[]` в полях разрешена (`{адрес: {улица: Строка}}`, `{заказы: Заказ[]}`); имя требует только `#`.

## Нотация литералов

Литерал записывает значение в одном из двух режимов. **Значение** (без тега) — тип известен из контекста (поле, переменная, элемент контейнера): пишем значение без тега. **Селектор** — литерал стоит отдельно и потому **несёт свой тип**, называя его. Базовые скаляры (`Число`, `Булево`, `Строка`, `Пусто`) самоочевидны из формы и тега не требуют.

Тег-тип в селекторе ставится по форме значения: **атомарное** значение (скаляр, ссылка) тег берёт в `(…)`; **структурное** (кортеж, отношение) уже имеет свои скобки — тег идёт **префиксом**, сохраняя делимитер (ср. C#: `new Point{…}`, `new Order[]{…}`).

| Тип | Значение (в контексте) | Селектор (отдельно) |
|---|---|---|
| Число | `5`, `-3.2` | — (базовый) |
| Булево | `true`, `false` | — (базовый) |
| Строка | `"Пятёрочка"` | — (базовый) |
| Пусто | `null` | — (базовый) |
| Регэксп → `Строка[]` | `r".+@.+"` | — |
| Дата | `"2024-04-23"` | `Дата("2024-04-23")` |
| Время | `"15:00"` | `Время("15:00")` |
| Положительное (подтип) | `5` | `Положительное(5)` |
| Кортеж | `{x: 3, y: 5}` | `Точка{x: 3, y: 5}` |
| Отношение | `[ {…}, {…} ]` | `Заказ[ {…}, {…} ]` (пустое — `Заказ[]`) |
| Внешний ключ | `"…uuid…"` | `#Заказ("…uuid…")` |

Тег отношения — это тип **элемента** (`Заказ`), а `[…]` даёт «отношение из». Внешний ключ — ссылка по значению (на `#` цели), а не указатель: `#Заказ` называет целевой тип, строка — значение `#`.

Формальная грамматика — в разделе «Сводная грамматика». Разбор однозначен: `имя` вплотную к `{` / `[` / `(` начинает селектор.

**Внутри контейнера литерал уточнять не надо — его тип выводится из описания.** Поля кортежа и кортежи отношения пишутся без тега, как значения. Так, переменная `Заказы` типа `Заказ[]` содержит:

```
[
  {объект: "Машиностроителей 15", дата: "2024-04-23", количество: 2},
  {объект: "Ленина 1",            дата: "2024-04-24", количество: 5}
]
```

Тип кортежей известен из `Заказ[]`, поэтому внутри `дата` (строка → `Дата`) и `количество` (`2` → `Количество`) без тега — их типы из `Заказ`; видно, что все кортежи одной шапки, а не «любые». Внешний ключ внутри тоже без тега — `заказ: "…uuid…"` (тип `#Заказ` из описания).

### Совместимость с JSON

Форма `значение` — **надмножество JSON**: любой JSON-документ является литералом. Для этого нотация значений ослаблена относительно реляционного ядра:

- **`null`** — допустимое значение: **единственное значение системного домена `Пусто`** (`x: null`, `[null]`). Реляционная модель (TTM — Кодд, Дейт, Дарвен) null *отвергает* (проскрипция NO NULLS), но essensio от неё **сознательно отходит ради практики**: JSON — основной формат ввода и несёт `null` повсюду, моделировать отсутствие декомпозицией непрактично. Поэтому `null` — полноценный член домена `Пусто`, а не «дыра» вне системы. Цена размена честная: `Пусто` несёт лишь факт «значения нет» и не подставляется в операции других доменов — сравнение с ним пишут явно.
- **число с экспонентой** — `1e3`, `-1.5E-2` (полный диапазон JSON-чисел);
- **пустой объект** `{}` — кортеж-значение из нуля полей;
- **строковый ключ** — ключ пары может быть строкой (`{"order-id": 1}`), а не только именем; так выражается JSON-объект с произвольными ключами.

Цена ослабления честная: `null` теперь — значение домена `Пусто` (выше), а вот прочие послабления (упорядоченный/разнородный массив; объект-словарь) дают **сырые данные**, а не значение домена. Сырьё типизируется лишь там, где из описания виден подходящий тип; иначе остаётся литералом-документом.

## Сводная грамматика

Полная EBNF вынесена в **единственный нормативный** файл [`grammar.ebnf`](../grammar.ebnf): все продукции в одном месте (разделы выше дают грамматику в прозе, примерах и семантических заметках, без повтора EBNF). Блоки связаны общими нонтерминалами `имя`, `тип`, `значение`, `выражение`. Три уровня: **тип** (что допустимо в ячейке) · **значение** (константа-литерал; тип из контекста, из селектора, либо домен `Пусто` у `null`) · **выражение** (вычисляемое); литерал — константный фрагмент выражения.

**Согласованность.**

- Общие нонтерминалы определены однократно: `имя` (лексика), `тип` (типы), `значение` / `выражение` (значения); остальные блоки на них ссылаются.
- Структуры различены по уровню: `кортеж-тип` / `отношение-тип` (типы компонент внутри) против `кортеж-знач` / `отношение-знач` (значения внутри) — одна форма скобок, разное содержимое.
- `литерал ⊂ выражение`: `атом` покрывает все формы `значение` / `селектор` плюс ссылку, применение операции, `_` и скобки; `Дата("…")` парсится обоими путями к одному значению — это согласовано.
- Граница: компоненты `кортеж-знач` и элементы `отношение-знач` — `значение` (константы) даже внутри выражения; построение кортежа/отношения из под-выражений — задача реляционной алгебры (открытый пункт).
